## 평가

### - 느낀점
이걸 언제 어떻게 쓸까란 생각으로 학원에서 막연하게 배웠던 쿼리 같은 지식들이, 웹 개발에 어떻게 쓰이는 방식을 알게 된 게 가장 놀랍고 신기했다. 무언가를 만들어봄으로써 머리속에 펼쳐져있던 파편들이 한 개념으로 뭉쳐지는 즐거운 경험이었다. 

### - 어려웠던 점
값을 불러오는 게 중요하다는 건 알지만 생각처럼 잘 되지 않아서 제일 어려웠다. 디버거와 크롬 개발자모드를 활용하라는 말을 들었지만 생소해서 제대로 사용하기까지 많이 버벅였다. 그래도 개발자모드를 자주 봤더니 css 하나는 제대로 보게 된 것 같다. 

### - 아쉬운 점 / 보완할 점
처음 프로젝트를 시작할 때는 남들과는 다르게 하고싶단 욕심이 컸다. 프로젝트부터 생성하고 부트스트랩이란 걸 써서 직접 뷰페이지도 바꾸며, 있던 DB 스키마나 서버도 새로 만들며 안 써본 깃허브까지 활용했다. 게시판 형태의 프로젝트가 과제였으니 그에 맞는 컨셉을 설정했고, 게시판이 네 개나 들어가니 각각의 게시판마다 데이터명이나 변수명, 파일명과 폴더명을 조금씩 바꾸어가면서 내가 제대로 이해한 게 맞는지 확인하려 했다. 때문에 제대로 이해는 했지만 개발환경을 조성하고 바로 개발을 들어가는 데 항상 시간이 걸렸다. 깃허브로 불러오는 파일 충돌로 서버단의 오류를 해결하는 데 기간의 40퍼센트를 보낸 것 같았다. 덕분에 오류를 많이 만나서 오류에 대한 두려움은 없지만 이런 과정이 없었으면 기능 몇 개는 더 추가할 수 있었을 거란 아쉬움이 컸다.
<br>
또한, 실험해본다고 이름을 다양하게 지정했더니 나중엔 헷갈려서 내가 원하는 파일을 찾는 과정 자체가 늦어졌다. 연습에서는 이해를 위해 필요한 과정이지만 프로젝트를 할 땐 직관적이고 통일된 이름이 필요하고 그에 대한 정리가 필요하다고 느꼈다.
<br>
기능 구현과 뷰 페이지를 신경 쓰느라 DB 설계에 대해 오래 생각하지 못한 것이 아쉬웠다. 사실 이전까지는 별 생각을 못했는데 댓글 기능이 추가되고 연관된 테이블이 많아지면서 설계에 대해 다시 생각해보게 되었다. 이런저런 기능을 넣고 싶어도 실력 부족으로 기능을 하나하나 구현하는 데에도 시간이 꽤 걸린 것도 아쉬웠는데, 그만큼 테이블 수가 적어서 DB 설계를 좀 더 자세히 하지 못한 게 제일 아쉽게 느껴졌다. 기능을 전부 구현하지 못하더라도 설계는 자세하게 해볼 수 있었을 텐데 싶다. 팀프로젝트를 들어갈 땐 빨리 컨셉 정하고 테이블 설계에 시간을 많이 투자해야겠다. 

### - 앞으로의 계획
부트스트랩으로 작업하면서 관련 지식이 많이 늘었다. 많이 접해서 그런 이유도 있겠지만 코드 정리와 주석의 힘이 가장 컸다고 느꼈다. 나만해도 들여쓰기가 조금만 많아도 정리가 되지 않았는데, 부트스트랩을 적용하면서 템플릿을 배포하기 위해 남들에게 가독성 있는 코드로 보이려고 얼마나 고민했는지가 엿보였고 반성했다. 나중에 같이 일할 팀원들을 위해서라도 평소에 주석이나 들여쓰기를 사용해 가독성있는 코드를 쓰려고 신경써야겠다.
<br>

배운 것을 언제 어디서 쓸 지 모르고, 모든 것을 기억하는 데는 한계가 있으니 바로 꺼내볼 수 있게 기록을 꾸준히 해야겠다고 느꼈다. 특히 오류노트의 경우엔, 고치고 바로 넘어갈 수 있는 문제인데 한번 기록함으로써 다음 오류에 대처할 수 있고, 오류를 겁내지 않는단 점에서 좋은 선택이었던 것 같다.
<br>

사족이지만 영어를 공부해야겠다고 느꼈다. 오류를 읽어내고 서치하는 과정에서 영어를 배우면 수월하겠단 생각을 많이 해서이다. 언젠가 개발 공부와 꼭 병행하고 싶다.
